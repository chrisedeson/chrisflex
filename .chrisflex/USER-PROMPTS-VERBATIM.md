It feels like the "amnesia" kicks in just when you're getting into a groove, doesn't it? The issue you're seeing—where the AI starts missing letters or small details—is usually a byproduct of Lossy Compression.

Here is the breakdown of why these agents "forget" and why they specifically start making those "typo-like" errors.

1. The "Compaction" Bottleneck
When your conversation history (the context) hits a certain limit (e.g., 200k tokens), Claude Code and Copilot don't use a process called Compaction or Summarization.

How it works: The AI takes the last 50 messages and rewrites them into a single summary block.

The "Letter" Problem: Summaries are meant to capture intent, not syntax. If the summary says "We refactored the user_auth function to use JWT," the AI might forget that you used a specific, non-standard variable name like uAuth_jwt_token. When it tries to write code again, it "guesses" the name based on the summary, leading to those missing letters or slight naming errors.

2. Claude Code's "200-Line" Rule
Claude Code has a very specific technical quirk that often causes this "forgetting" behavior:

The Limit: It loads the first 200 lines of your MEMORY.md file into its system prompt automatically.

The Result: If you've been working a long time and your auto-learned patterns or rules move to line 201, Claude effectively "forgets" them. It still knows the file exists, but it isn't "looking" at it anymore unless you explicitly tell it to read that file again.

3. The "Context Rot" Phenomenon
Even before a window is "full," AI models suffer from Attention Dilution.

In a massive context window, the model's "attention" is spread thin across thousands of tokens.

Research shows that models are great at remembering the beginning and end of a prompt but struggle with the "middle." As your session grows, the important technical details (like specific character requirements) get buried in the middle of the "stack," causing the agent to start hallucinating or skipping characters.

4. How to Stop the "Forgetting"
To prevent those "missing letter" errors, you have to force the agent to stop relying on its "fuzzy" summary memory:

Strategy	Action
Manual Sync	Periodically use the /compact command yourself. This forces a clean summary while you still have the mental context to correct it if it misses a detail.
The "Sidebar" Trick	Keep your CLAUDE.md or copilot-instructions.md open in a split-tab. Both agents give higher priority to "open files" than to "old chat history."
Rule of One	Instead of one massive file, create small reference files (e.g., AUTH_STANDARDS.md). Claude Code is better at "reading on demand" than "remembering forever."
Reset the Host	In VS Code, use Ctrl+Shift+P -> Developer: Restart Extension Host. Sometimes the "forgetting" isn't the AI—it's the extension running out of RAM on your machine.

How GSD "Solves" the Forgetting Problem
The core philosophy of GSD is "Fresh Context = Clear Thinking." Here is the technical breakdown of how it manages memory:

1. Isolated Sub-Agents (The "Clean Slate" Strategy)
Standard Claude Code sessions get "polluted" as you chat. GSD stops this by spinning up ephemeral sub-agents for specific tasks.

The Problem: In a normal 3-hour session, the AI is carrying the weight of every mistake, every long log file, and every "Oops, wait" you've typed.

The GSD Fix: It breaks your project into tiny phases. Each phase runs in a brand-new sub-agent with a fresh 200k token window. It only hands that sub-agent the specific files and instructions it needs for that one task. When the task is done, that sub-agent is killed, and the next one starts fresh.
+2

2. Spec-Driven Architecture
GSD replaces "Chat Memory" with "File-Based Memory."

Before any code is written, GSD forces an "interrogation" phase where it builds a .planning/ directory.

This directory contains the Source of Truth (files like ROADMAP.md, CONVENTIONS.md, and TASKS.md).

Instead of "remembering" your project, the AI reads the spec at the start of every task. It's like giving a person a checklist every 5 minutes so they never have to rely on their memory.

3. The "3-Task" Limit
GSD enforces a rule where a plan can have at most 3 tasks.

By keeping the work window small, the agent never hits the "Compaction" phase where it starts summarising (and losing those specific letters/variable names you mentioned).

Each task usually ends with an atomic git commit, so if it does mess up, the system can revert and try again with a clean state.

4. Parallel Mapping
When you start a project, GSD doesn't just "look around." It launches parallel agents (usually 4 at once) to map your tech stack, architecture, and technical debt. They write their findings into markdown files, ensuring the AI "knows" your code structure through documentation rather than vague chat history.
